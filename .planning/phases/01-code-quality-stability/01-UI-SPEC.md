## Scope Note

Phase 1 is a hardening phase. The CONTEXT.md is explicit: "Keine neuen Features, keine UI-Änderungen — Phase 2 übernimmt das UI." The design contract here covers **only the three new interaction surfaces introduced in Phase 1**:

1. `QuotaExceededError` banner (ERR-02)
2. JSON import failure banner (TYPES-01 / existing pattern)
3. Snapshot naming input at creation time (SNAP-01)
4. T-shortcut new-node auto-focus flow (SNAP-02)

Everything else remains exactly as-is. Phase 2 owns the full visual redesign.

## Spacing Scale

Existing project uses Tailwind's default scale (4px base unit). Phase 1 new elements follow this scale exactly.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, tight inline spacing |
| sm | 8px | Compact element padding (e.g. snapshot list item `py-1`) |
| md | 16px | Default element spacing |
| lg | 24px | Section padding |
| xl | 32px | Layout gaps |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

Exceptions:
- Snapshot name input: `px-2 py-1` (8px horizontal, 4px vertical) — matches existing `Button` compact style
- Banner: `px-3 py-2` (12px / 8px) — matches existing `Banner.tsx` pattern exactly; do not change

Source: existing `Banner.tsx` and `Button.tsx` Tailwind classes; default Tailwind scale.

## Component Inventory (Phase 1 new elements only)

### 1. QuotaExceeded Banner

Reuse existing `Banner.tsx` component without modification. New state variable `quotaError: string | null` in `GraphCanvas.tsx`. Render `<Banner>{quotaError}</Banner>` when non-null. Clear on next successful save.

```tsx
// Pattern — matches existing Banner usage
{quotaError && <Banner>{quotaError}</Banner>}
```

Visual: `bg-yellow-50 text-yellow-900 border border-yellow-200 rounded px-3 py-2 text-sm` — identical to existing banner.

### 2. Snapshot Name Input

Inline text input added to the "+ Neu" button flow inside `AppToolbar.tsx` snapshot panel. Input appears **before** the existing "+ Neu" button in the panel header row. Submitting (Enter key or clicking "+ Neu") creates the snapshot with the entered name.

```tsx
// Layout: [name input] [+ Neu button] in flex row, gap-2
// Input: text-xs px-2 py-1 border rounded w-full max-w-[140px]
// Placeholder: "Name (optional)"
```

Visual spec:
- `text-xs` (12px) to match surrounding snapshot list text
- `border border-neutral-300 rounded px-2 py-1`
- `focus:outline-none focus:ring-1 focus:ring-blue-500`
- `placeholder:text-neutral-400`
- No label element; placeholder is sufficient (field is optional)

### 3. T-Shortcut Node Auto-Focus

No new visual component. Existing `TaskNode.tsx` receives `focusOnMount?: boolean` in its node data (per RESEARCH.md open question resolution). The `TaskNode` calls `titleInputRef.current?.focus()` in a `useEffect` when `focusOnMount === true`. The prop is cleared from node data after focus triggers to prevent re-focus.

Visual: No change to node appearance on creation. Focus ring on the title input uses the existing `focus:ring-2 focus:ring-blue-500` pattern from `Button.tsx`.

## Accessibility

- Snapshot name input: no `<label>` required — placeholder is `"Name (optional)"` and the input is decorative/optional. If WCAG AA is required post-Phase-2 audit, a visually-hidden label can be added then.
- Error banners: rendered as `<div>` (existing `Banner.tsx`). No `role="alert"` in Phase 1 — adding ARIA roles is a Phase 2 concern.
- T-shortcut: keyboard shortcut `T` must not fire when focus is in any `HTMLInputElement` or `HTMLTextAreaElement`. This is implemented via the guard in SNAP-02 (per RESEARCH.md).
- Touch targets: all interactive elements in Phase 1 use existing `px-3 py-1.5` (Button) or `px-2 py-1` (compact inputs) — no new touch targets added.
